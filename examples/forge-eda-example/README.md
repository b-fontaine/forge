# forge-eda-example

A **reference project** for the Forge `event-driven-eu` archetype: an
EU-sovereign, event-driven Rust backend — a NATS JetStream event backbone,
a Postgres append-only event store, and Temporal activity-only saga
orchestration, with AsyncAPI 3.1 as the event single source of truth. It is
the event-driven sibling of
[`forge-fsm-example/`](../forge-fsm-example/) (`full-stack-monorepo`) and
[`forge-rag-example/`](../forge-rag-example/) (`ai-native-rag`).

Adopters can inspect every artefact — the rendered project tree, three
archived demo changes (proposals, specs, designs, ADRs, tasks, `.feature`
files) — directly on GitHub without cloning.

> **Status — promoted archetype.** `event-driven-eu` is `stage: stable`
> (`scaffoldable: true`), promoted by `b6-7-harness` after a green ≥35-test
> suite + the live `harness-rust` codegen/build gate. This tree was
> therefore rendered by the **real** `forge init --archetype
> event-driven-eu` CLI — the exact command a real adopter runs — **not**
> via an `overlay.sh` workaround (which is what `forge-rag-example` had to
> use because `ai-native-rag` was still a `candidate` at its creation time).
> The committed `.forge/scaffold-manifest.yaml` records the exact
> scaffold-time version (`archetype_version: 1.0.0`).

## How this example was built

The tree was generated by running the real Forge CLI —
`forge init --archetype event-driven-eu` — on a clean directory, then
committed verbatim:

```bash
forge init forge-eda-example \
  --archetype event-driven-eu \
  --org io.forge.example
```

`forge init` resolves the archetype through the dispatch table
(`.forge/scaffolding/dispatch-table.yml`), confirms the schema is
`stage: stable` / `scaffoldable: true` (`selectScaffoldableVersion` returns
`1.0.0`), and delegates to the per-archetype wrapper
`bin/forge-init-event-driven-eu.sh`. The wrapper renders the archetype's 48
templates through `overlay.sh`, writes `.forge/scaffold-manifest.yaml` (with
the plan SHA, the template-set SHA, and the scaffold date), then runs
best-effort `buf generate` (Connect/gRPC + AsyncAPI derived stubs) and
`cargo fetch` — both skipped gracefully when the buf registry / crates.io are
unreachable (the render's success contract is the tree, not codegen).

Because the archetype is promoted, this example proves the *public adopter
flow* end-to-end: `forge init` really does scaffold a complete, buildable
event-driven project.

After rendering, this example adds its **own** `.forge/` framework assets
(constitution, standards, schemas, and the deterministic gate scripts
`verify.sh` + `constitution-linter.sh`) so the tree self-validates
standalone, exactly as `forge-fsm-example/` and `forge-rag-example/` do.

## Demo changes

Three **archived** demo application changes live under
[`.forge/changes/`](.forge/changes/) (see
[`.forge/changes/MANIFEST.md`](.forge/changes/MANIFEST.md) for the
chronological index). Each carries the full Forge lifecycle — proposal, specs
delta, design (per-layer for the multi-layer demo), tasks, and a `.feature`
file — mapped onto the rendered backend modules (which ship inline
`#[cfg(test)]` tests).

| Demo | Layers | What it demonstrates |
|---|---|---|
| [`demo-001-ingestion-http-nats`](.forge/changes/demo-001-ingestion-http-nats/) | `[backend]` | Ingestion HTTP → NATS: an axum endpoint wraps a command in a versioned, idempotent `EventEnvelope` and publishes it to NATS JetStream with `Nats-Msg-Id` dedup. Event versioning + idempotency keys. cucumber-rs BDD. |
| [`demo-002-projection-readmodel`](.forge/changes/demo-002-projection-readmodel/) | `[backend]` | Projection: a consumer folds the persisted event stream (Postgres event store) into a deterministic, replayable read-model projection, guarded by the inbox dedup (outbox/inbox pattern). cucumber-rs BDD. |
| [`demo-003-order-saga`](.forge/changes/demo-003-order-saga/) | `[backend, infra]` | Saga (3 steps): a multi-layer (Janus-orchestrated) Temporal **activity-only** saga — reserve stock → charge payment → confirm shipment — with reverse-order compensation on failure (Article VIII.2). Per-layer designs/ + tasks/ (backend saga crate + infra Temporal cluster). |

## What this example does NOT show

- **No ops-console frontend.** The archetype's `frontend` layer is a single
  DEFERRED `ops-console` surface (`ADR-B6-1-004`) — this archetype is
  backend-centric, so the example ships no web UI. The multi-layer (Janus)
  demo therefore spans `[backend, infra]`, not `[backend, frontend]`.
- **No live NATS / Temporal / Postgres calls.** The demos exercise the
  pipeline with the in-memory fakes the archetype ships
  (`InMemoryEventStore`, the recording publisher, the in-process saga
  coordinator); nothing reaches a live broker, database, or Temporal server.
- **No compliance demo.** NIS2 incident reporting / DORA RoI / SBOM
  CycloneDX arrive in `b6-9-compliance`; this example references the
  EU-sovereignty posture but ships no compliance demo.
- **No Janus-rule refusal demo.** The event-broker refusals (no Confluent
  Cloud / no US-managed Kafka SaaS) are shipped by `b6-10-janus-rule`.

## Reproducing this example

The archetype is `stable` / `scaffoldable`, so the real CLI renders it —
this is the exact command used to build the tree:

```bash
# From a clone of the Forge framework repo (CLI built via `cd cli && npm ci && npm run bundle`):
forge init forge-eda-example \
  --archetype event-driven-eu \
  --org io.forge.example \
  --target /tmp/forge-eda-example
# (equivalently, before installing the CLI on PATH:
#  node cli/dist/index.js init forge-eda-example \
#    --archetype event-driven-eu --org io.forge.example \
#    --target "$PWD/examples/forge-eda-example" --force)
```

That produces the `backend/`, `infra/`, `shared/asyncapi/`,
`shared/protos/`, `.github/workflows/` tree and the
`.forge/scaffold-manifest.yaml` you see committed here. The example's own
`.forge/` framework assets (constitution, standards, schemas, gates) and the
three `.forge/changes/` demos are added on top.

Validate the tree's own gates:

```bash
cd /tmp/forge-eda-example     # or this committed example directory
bash .forge/scripts/verify.sh
bash .forge/scripts/constitution-linter.sh
```

> **Why these examples are excluded from Forge's own gates.** The framework's
> deterministic gates ship with skip-guards (`FR-GL-026`, `FR-GL-027`) that
> detect when they run from the Forge framework repo and exclude `examples/`.
> Each example owns its own gates; when a PR touches `examples/**`, the
> framework's `forge-ci.yml` `example` job invokes each example tree's gates.
