# forge-rag-example

A **reference project** for the Forge `ai-native-rag` archetype: a Rust backend
(RAG pipeline + in-repo LLM gateway proxy + MCP servers) paired with a Qwik
streaming web surface, backed by Postgres/pgvector (HNSW). It is the RAG sibling
of [`forge-fsm-example/`](../forge-fsm-example/), which demonstrates the
`full-stack-monorepo` archetype.

Adopters can inspect every artefact — the rendered project tree, three archived
demo changes (proposals, specs, designs, ADRs, tasks, `.feature` files), and the
AI-First materialisation — directly on GitHub without cloning.

> **Status — candidate archetype.** `ai-native-rag` is `stage: candidate`
> (`scaffoldable: false`), so `forge init --archetype ai-native-rag` refuses
> cleanly (exit 3). This tree was therefore rendered via `overlay.sh` (the
> renderer of record for this archetype, `ADR-B7-2-007`), **not** via `forge
> init`. Promotion to `stable` is gated on a green `b7-6-harness`. The committed
> `.forge/scaffold-manifest.yaml` records the exact scaffold-time version
> (`archetype_version: 1.0.0`, `stage: candidate`) so you know precisely what
> you are inspecting.

## How this example was built

The tree was generated by rendering the `ai-native-rag/1.0.0` scaffold-plan
through `overlay.sh` — **not** through `forge init` (which refuses while the
archetype is `candidate`). The exact reproduction is in
[Reproducing this example](#reproducing-this-example).

`overlay.sh` is the pure-Python renderer that `init.sh` calls internally; it
iterates the archetype's `scaffold-plan.yaml` `templates[]` list, substitutes the
`<project-name>` / `<reverse-domain>` / `<root-module>` placeholders, and writes
`.forge/scaffold-manifest.yaml` with the plan SHA, the template-set SHA, and the
scaffold date. Rendering through `overlay.sh` makes the committed tree provably
the templates' output — the same renderer `b7-2-scaffolder` validated the
archetype with.

After rendering, this example adds its **own** `.forge/` framework assets
(constitution, standards, schemas, and the deterministic gate scripts
`verify.sh` + `constitution-linter.sh`) so the tree self-validates standalone,
exactly as `forge-fsm-example/` does.

## What this example does NOT show

- **It is not scaffoldable yet.** `ai-native-rag` is `candidate` /
  `scaffoldable: false`; `forge init --archetype ai-native-rag` refuses (exit 3)
  until `b7-6-harness` promotes it. This tree was rendered via `overlay.sh`.
- **No Flutter mobile surface.** The archetype is web-public only
  (`ADR-B7-2-006`); the only frontend is the Qwik web surface.
- **No AI-Act / DORA compliance demo.** Those artefacts arrive in
  `b7-5-ai-act`; this example references `compliance-tiers` where the gateway's
  tier-aware hooks appear, but ships no compliance demo.
- **No Pythia agent demo.** The product-analyst RAG agent is a separate brick
  (`b7-pythia`).
- **No live LLM upstream calls.** The demos exercise the pipeline with a
  local/fake embedder and a stubbed gateway upstream; nothing reaches a real
  model. (demo-003 **does** show the streaming UX via b7-10's
  `QueryStream`/`queryStream` contract.)

## Reproducing this example

The archetype is `candidate`, so `forge init` will refuse. Render the tree the
same way this example was built — directly via `overlay.sh`:

```bash
# From a clone of the Forge framework repo:
ARCHETYPE_DIR="$PWD/.forge/templates/archetypes/ai-native-rag"

# overlay.sh hard-codes its archetype dir to the flagship, so absolutize the
# ai-native-rag plan's relative source: paths into a temp plan first.
absplan="$(mktemp)"
ARCHETYPE_DIR="$ARCHETYPE_DIR" python3 - "$absplan" <<'PY'
import os, sys, yaml
arch = os.environ['ARCHETYPE_DIR']
plan = yaml.safe_load(open(os.path.join(arch, 'scaffold-plan.yaml')))
for e in plan.get('templates', []):
    s = e.get('source', '')
    if s and not os.path.isabs(s):
        e['source'] = os.path.join(arch, s)
yaml.safe_dump(plan, open(sys.argv[1], 'w'), sort_keys=False)
PY

bash .forge/scripts/scaffolder/overlay.sh \
  --target /tmp/forge-rag-example \
  --project-name forge-rag-example \
  --reverse-domain io.forge.example \
  --root-module forge_rag_example \
  --plan "$absplan" --force
rm -f "$absplan"
```

That produces the `backend/`, `frontend/web-public/`, `infra/`, `shared/protos/`
tree and the `.forge/scaffold-manifest.yaml` you see committed here. The
example's own `.forge/` framework assets (constitution, standards, schemas,
gates) and the three `.forge/changes/` demos are added on top.

Validate the tree's own gates:

```bash
cd /tmp/forge-rag-example     # or this committed example directory
bash .forge/scripts/verify.sh
bash .forge/scripts/constitution-linter.sh
```

> **Why these examples are excluded from Forge's own gates.** The framework's
> deterministic gates ship with skip-guards (`FR-GL-026`, `FR-GL-027`) that
> detect when they run from the Forge framework repo and exclude `examples/`.
> Each example owns its own gates; when a PR touches `examples/**`, the
> framework's `forge-ci.yml` `example` job invokes each example tree's gates.
