# Proposal: demo-002-greeting-screen

<!-- Audit: C.1 (illustrative demo) -->
<!-- Layers: [frontend] — single-layer demo -->
<!-- Depends on: demo-001-greeting-service (provides the Greet RPC) -->

## Problem

Adopters need a concrete demonstration of how a **single-layer
frontend change** flows through the Forge pipeline — Flutter
Clean Architecture + Feature-Sliced Design, BLoC state
management, widget tests, golden tests, and BDD via
`bdd_widget_test`.

## Solution

Add a minimal `GreetingScreen` Flutter widget under
`frontend/lib/features/greeting/` that lets the user enter a
name in a text field, taps "Say hello", and displays the
greeting message returned by the cubit. The cubit
(`GreetingCubit`) consumes the gRPC `Greet` RPC defined by
demo-001-greeting-service.

This demo is **deliberately trivial** — its purpose is to
demonstrate the full Flutter discipline (Article VI) end-to-end,
not to ship a real product feature.

## Scope In

- A `GreetingCubit` with three states (`initial`, `loading`,
  `success`) under `frontend/lib/features/greeting/presentation/cubit/`.
- A `GreetingScreen` widget under
  `frontend/lib/features/greeting/presentation/screen/`.
- A `GreetingRepository` port (interface) in the domain layer.
- A `GreetingRepositoryImpl` data adapter that calls the gRPC
  client (in this demo : a fake in-memory implementation, since
  the proto stubs are not generated by the scaffolder ; a real
  gRPC client wiring is left as a follow-up).
- Widget test covering the initial → loading → success
  transition.
- BDD scenario in `features/greeting_screen.feature`.

## Scope Out

- No actual gRPC wiring (left as a follow-up — the demo focuses
  on Flutter discipline, not stub generation).
- No persistence of greeting history.
- No internationalization (English only).
- No navigation routing changes — the screen is a standalone
  widget.

## Impact

- **Technical impact**: Small. ~80 lines of Dart split across
  cubit / repository / widget + ~40 lines of widget test.
- **Dependencies**: `flutter_bloc`, `bloc_test`, `mocktail`,
  `bdd_widget_test` (added to `pubspec.yaml`).
- **Risk level**: Trivial. The demo is illustrative.

## Constitution compliance

- **Article I (TDD)**: every cubit transition is exercised by a
  test written RED-first.
- **Article II (BDD)**: `features/greeting_screen.feature`
  declares the user-facing scenario in Gherkin.
- **Article III (Specs Before Code)**: this proposal precedes
  the spec which precedes the design which precedes the code.
- **Article VI.2 (Clean Architecture + FSD)**:
  presentation / domain / data triad under
  `lib/features/greeting/`.
- **Article VI.3 (BLoC)**: state via `flutter_bloc::Cubit`.
- **Article VI.8 (Golden tests)**: the screen ships with one
  golden test rendering the initial state in light theme.
- **Article VI.9 (Accessibility)**: the text field and submit
  button carry semantic labels.
