## Problem

A new adopter evaluating Forge has **no concrete artefact** to
inspect. The audit roadmap explicitly identifies this as a
deal-breaker for adoption :

> Le nouvel utilisateur n'a aucun moyen de voir à quoi ressemble
> une **bonne** proposal, un **bon** specs.md delta, un **bon**
> design avec ADRs.

The Forge repo itself contains 5 archived changes
(`b1-foundations`, `b1-scaffolder`, `b1-workflow`, `b1-delivery`,
`g1-forge-ci`) which model the discipline well — but they are
**meta-changes** (Forge improving Forge), not application changes
in a downstream project. An adopter wants to see :

1. What does a freshly-scaffolded project look like ? (the **shape**)
2. What do realistic **archived application changes** look like
   (proposal → specs → design → tasks → archive) for typical
   features like "add user authentication", "add a CRUD endpoint",
   "add a Flutter screen with tests" ?
3. How do the per-layer FRs (`FR-BE-NNN`, `FR-FE-NNN`,
   `FR-IN-NNN`) flow through `Janus` orchestration in a real
   multi-layer change ?
4. How does the local OTel + SigNoz stack behave under traffic
   from a real backend + frontend ?
5. What does `forge-ci.yml`-equivalent CI look like when actually
   running ? (validates the four reference workflows shipped by
   `b1-delivery`)

Without a concrete reference, every claim Forge makes is an
abstraction. The 75/75 PASS test count and the 17 commits since
v0.2.1 are great signals — but they are **internal validation**.
External validation requires a project an adopter can clone,
read, and reproduce.

This change closes the **last big T2 gap** flagged by the audit
roadmap : a public reference project that demonstrates the full
Forge pipeline end-to-end.

## Solution

Build a **reference project** under `examples/forge-fsm-example/`
in the Forge repository, scaffolded for real via
`/forge:init --archetype full-stack-monorepo`, with **3 archived
demo application changes** + **1 active change at status
`specified`** to demonstrate every phase of the pipeline.

### Why `examples/` inside the Forge repo (and not a separate repo)

Three options were considered :

| Option | Description | Trade-offs |
|---|---|---|
| **A — separate repo** (`forge-fsm-example` standalone) | Most accurate to the audit's intent. Adopters clone it independently. | Requires creating + maintaining a second repo, second CI, two-way sync when archetype evolves. Heavy ops surface for the maintainer. |
| **B — `examples/` in Forge** *(recommended)* | Tracked alongside the framework. Always in sync with the archetype version. | Risk of `verify.sh` / `constitution-linter.sh` walking into `examples/` and getting confused. Mitigated by explicit skip-guards (small change to both scripts). |
| **C — generated on demand via script, not committed** | Smallest commit footprint. | Defeats the purpose : adopters can't just browse `.forge/changes/<name>/` on GitHub to read good specs. The whole point is to make artefacts inspectable. |

**Option B is recommended.** The maintenance argument outweighs
the ops complexity of a separate repo. Co-locating the example
with the framework guarantees the example demonstrates the
*current* archetype contract ; archetype bumps mechanically
require an example update in the same change cycle.

### What ships

1. **`examples/forge-fsm-example/`** — a fully-scaffolded
   `full-stack-monorepo` project tree, generated by running the
   actual `forge init --archetype full-stack-monorepo
   forge-fsm-example` command (per `b1-scaffolder`'s contract).
   Includes : `.forge/`, `.claude/`, `.mcp.json`, `frontend/`
   (Flutter), `backend/` (Rust workspace), `infra/` (k8s + kong
   + observability + docker-compose), `shared/protos/`,
   `.github/workflows/` (the 4 reference workflows from
   `b1-delivery`), root `CLAUDE.md`, `Taskfile.yml`,
   `docker-compose.dev.yml`, `.env.example`, `.gitignore`.

2. **3 archived demo application changes** under
   `examples/forge-fsm-example/.forge/changes/` :
   - **`demo-001-greeting-service`** (single-layer, backend only) —
     a minimal gRPC `Greeter` service in
     `backend/crates/grpc-api/`, with a domain entity in
     `backend/crates/domain/`, full TDD cycle (unit + integration
     tests), proto contract in `shared/protos/v1/greeting/`.
     **Demonstrates** : single-layer change workflow,
     proto-first design, hexagonal architecture, BDD scenario.
   - **`demo-002-greeting-screen`** (single-layer, frontend only) —
     a Flutter screen consuming the gRPC `Greeter` from demo-001,
     with widget tests + golden test, BLoC state management,
     a11y compliance.
     **Demonstrates** : Flutter clean+FSD architecture, generated
     proto consumption from `frontend/lib/generated/protos/`,
     widget testing patterns.
   - **`demo-003-rate-limit`** (multi-layer, backend + infra) —
     adds rate-limiting via Kong plugin (infra) consumed by the
     existing greeting service (backend). Triggers the **Janus**
     cross-layer orchestrator (≥ 2 layers).
     **Demonstrates** : multi-layer change with `layers: [backend,
     infra]`, per-layer designs (`design-backend.md`,
     `design-infra.md`), per-layer tasks, Janus dispatch.

3. **1 active demo change at status `specified`** :
   - **`demo-004-user-onboarding`** (multi-layer, backend +
     frontend + protos) — full user onboarding flow, currently at
     spec phase. **Demonstrates** : a real `[NEEDS CLARIFICATION]`
     entry being resolved through the spec lifecycle (open
     question → resolved → integrated).

4. **`examples/forge-fsm-example/README.md`** — top-of-tree
   navigation. Walks through how the example was built (which
   commands, in which order), where to look for what (specs,
   design, ADRs, tests). Links to each demo change with a
   one-sentence summary.

5. **`examples/README.md`** — meta-documentation. Explains the
   directory's purpose in the Forge repo, links to the canonical
   archetype documentation, points to `forge-fsm-example` as the
   first (and currently only) example.

6. **Skip-guards in Forge gates** :
   - `.forge/scripts/verify.sh` gains an early-exit on every
     section if `$FORGE_ROOT == <forge-repo-root>` AND the path
     being inspected falls under `examples/` — never recurse into
     the example tree. The example has its own `.forge/` and runs
     its own `verify.sh` independently.
   - `.forge/scripts/constitution-linter.sh` gets the same guard.
   - `.forge/scripts/tests/foundations.test.sh` and friends
     remain untouched (they walk specific dirs they own).
   - The Forge repo's `.gitignore` adds `examples/*/cli/` and
     `examples/*/node_modules/` to keep build artefacts out
     (Flutter `build/`, Rust `target/`, etc. are already covered
     by the per-language `.gitignore` inside each scaffolded tree).

7. **Example CI integration** :
   - `forge-ci.yml` (Forge's own CI from `g1-forge-ci`) gains
     a new job `example` that, on PRs touching `examples/**`,
     runs the 4 reference workflows of the example
     (`forge-backend.yml`, `forge-frontend.yml`, etc.) against
     the example tree, validating that the example is itself
     gate-passing. Skipped on PRs not touching `examples/**`.

8. **NFR validation via real metrics** :
   - The example's `task dev` invocation gives us a real measurement
     of NFR-015 (≤90s observability stack startup).
   - The example's CI runtime gives us a real measurement of
     NFR-013 (≤8min warm) and NFR-014 (≤30min integration).
   - Real `kustomize build overlays/dev` and `overlays/prod` outputs
     give us a real measurement of NFR-017 (≤4KB diff).
   - These numbers get appended to `standards/infra/k8s-overlays.md`
     and `standards/infra/observability-local.md` as observed
     baselines (replacing the "aspirational" labels in the
     b1-delivery NFRs).

## Scope In

- The fully-scaffolded `examples/forge-fsm-example/` tree
  (generated via the real `forge init --archetype
  full-stack-monorepo` command, then committed verbatim).
- 3 archived demo changes (`demo-001`, `demo-002`, `demo-003`)
  with all their lifecycle artefacts (proposal, specs, design,
  tasks, features, archive).
- 1 active demo change (`demo-004`) at status `specified` with
  realistic spec content (no design or tasks yet).
- `examples/README.md` + `examples/forge-fsm-example/README.md`.
- Skip-guards in `verify.sh` and `constitution-linter.sh` for
  the `examples/` tree.
- Forge `.gitignore` additions for example build artefacts.
- A new `example` job in `forge-ci.yml` that validates the
  example tree on PRs touching `examples/**`.
- NFR baselines appended to `standards/infra/*.md` based on
  real measurements from the example.
- Update to `.forge/specs/full-stack-monorepo.md` Scope section
  to reference the example as the canonical demonstration.
- Update to `docs/CONTRIBUTING.md` "Roadmap" section to mark
  C.1 done.
- Test harness `c1.test.sh` validating example structure
  (manifest.yaml present, demo changes follow expected naming,
  every demo has the 5 required artefacts).

## Scope Out (Explicit Exclusions)

- **No external GitHub repo** for the example — Option A is
  rejected. If a future change wants a standalone repo (for
  marketing or CI badge purposes), it can extract from the
  in-repo example.
- **No actual product semantics**. The "greeting service",
  "greeting screen", "rate-limit" features are deliberately
  trivial. The example's value is the **process artefacts**, not
  the product.
- **No `forge demo` CLI subcommand** that scaffolds an example
  on demand. Future enhancement, out of scope.
- **No video walkthrough or interactive tutorial** (audit module
  C.2 — separate change).
- **No anti-patterns gallery** (audit module C.3 — separate change).
- **No comparison matrix** with BMAD/SpecKit/Agent OS V3 (audit
  module C.4 — separate change).
- **No migration guide** from ad-hoc projects (audit module C.5
  — separate change).
- **No mobile-only or flutter-firebase reference projects** —
  those archetypes (B.2, B.4) don't exist yet.
- **No public deployment** of the example to staging/prod
  Kubernetes — local dev only via `task dev`.
- **No external dependencies** on specific cloud providers.

## Impact

- **Users affected** : every Forge adopter (massive uplift in
  onboarding clarity), Forge maintainers (new example to
  maintain alongside archetype evolution).
- **Technical impact** : Large. ~50-100 new files under
  `examples/forge-fsm-example/`, ~20-30 new files under
  `.forge/changes/` of the example (per demo × 5 artefacts × 4
  demos), 2 small script edits (`verify.sh` + `constitution-linter.sh`),
  1 new harness `c1.test.sh`, 1 workflow job extension. The
  scaffolder code itself is unchanged ; we exercise it.
- **Dependencies** : `b1-delivery` archived (provides the
  archetype contract), `g1-forge-ci` archived (provides the CI
  conventions for the new `example` job).
- **Risk level** : **Medium**.
  - The example tree is large ; review burden non-trivial.
  - First time we ship something that depends on `flutter
    create` + `cargo new` + `buf` — if the example fails to
    build on a contributor's machine, it weakens the archetype
    promise. Mitigation : the new `example` job in CI catches
    this.
  - Skip-guards in `verify.sh` and `constitution-linter.sh`
    are a behaviour change of pre-existing scripts. Tests must
    cover both paths (skip when in example, do not skip when
    not).
  - Sub-option : split this change into 3 sequential Forge
    changes (`c1a-scaffold`, `c1b-demo-changes`, `c1c-walkthrough`)
    if review fatigue is a concern. Recommendation : single
    change if reviewer bandwidth allows ; split if not.

## Constitution Compliance

### Article I — TDD

The example itself ships TDD-conformant code (every Rust crate
and Flutter widget has tests). Every demo change has its own
RED→GREEN cycle archived. The new `c1.test.sh` harness follows
the manifest pattern (RED first against an empty `examples/`,
GREEN once the example tree exists).

### Article II — BDD

Every demo change ships a `features/<name>.feature` file with
realistic Gherkin scenarios. Demo-001 covers gRPC happy +
error paths ; demo-002 covers widget interaction + a11y ;
demo-003 covers rate-limit threshold + bypass ; demo-004's spec
already declares the BDD outline.

### Article III — Specs Before Code

Every demo change has a complete proposal → specs → design →
tasks pipeline. No demo's code was written before its spec.
The example **is** the demonstration of Article III.

### Article IV — Semantic Deltas

Each demo change's specs.md uses the ADDED / MODIFIED / REMOVED
delta format. Demo-003 (multi-layer) demonstrates per-layer
delta semantics in `specs.md` with FR IDs prefixed by layer
(`FR-BE-001`, `FR-IN-001`).

### Article V — Conformance Gate

The example's own gates (`verify.sh`, `constitution-linter.sh`,
the 4 reference workflows) all pass on every commit to the
example tree. The new `example` job in `forge-ci.yml` enforces
this in CI.

### Article VI — Flutter Architecture

Demo-002 follows Clean + FSD strictly : `lib/core/` (Riverpod
config, theming), `lib/shared/` (cross-feature widgets),
`lib/features/greeting/` (presentation/domain/data triad).
Widget tests + golden test + BDD scenario.

### Article VII — Rust Architecture

Demo-001 follows hexagonal strictly : `crates/domain/` (pure,
zero deps), `crates/application/` (use cases), `crates/grpc-api/`
(adapter consuming `tonic-build` from `shared/protos/`),
`crates/infrastructure/` (sqlx repo if needed), `bin-server/`
(wiring). Mockall for unit tests, BDD via cucumber-rs.

### Article VIII — Infrastructure

Demo-003 exercises the Kong rate-limit plugin pattern. The
example's `infra/` tree uses the canonical Kustomize overlays
from `b1-delivery`. Local docker-compose includes the OTel +
SigNoz stack ; the example demonstrates real traces.

### Article IX — Observability

The example's backend and frontend instrument themselves with
the canonical OTLP env vars from `backend/.env.dev` and
`frontend/.env.dev` (FR-IN-009 of `b1-delivery`). Real spans
emitted, viewable in SigNoz on `task dev`.

### Article X — Quality

Every demo change passes Nemesis (Flutter quality gate) for the
frontend, Tribune (Rust quality gate) for the backend. CI
enforces zero warnings.

### Article XI — AI-First

Out of scope for this example. A future demo could add an
AI-first demo change to exercise Article XI ; deferred.

## Open Questions for the design phase

The proposal makes definitive choices on the strategic frame.
Three minor decisions remain for the design phase :

1. **Whether to ship demo-004 at status `specified` with FR-only
   content, or escalate it to `designed` with ADRs to fully
   demonstrate the design phase too.** Recommendation :
   `specified` only — the design phase is illustrated by the 3
   archived demos that already shipped `design.md` files.

2. **Whether the new `example` job in `forge-ci.yml` runs the 4
   reference workflows from inside the example, or runs an
   adapter that walks them.** Recommendation : run them
   directly via `act` if available, fallback to a structural
   YAML parse + skip otherwise.

3. **Whether to split this change into 3 sub-changes** (c1a /
   c1b / c1c) for reviewer ergonomics. Recommendation : keep as
   one change, but ship in 3 commit clusters with clear
   boundaries (scaffold / demos / docs+CI).
