This change ships **one consolidated mono-change** (Option B + mono-change
per the proposal) that delivers, in a single archive cycle :

1. The fully-scaffolded reference project under
   `examples/forge-fsm-example/` (generated by the real
   `forge init --archetype full-stack-monorepo` command, then committed
   verbatim).
2. Three archived demo application changes (`demo-001-greeting-service`,
   `demo-002-greeting-screen`, `demo-003-rate-limit`) plus one active
   demo at status `specified` (`demo-004-user-onboarding`).
3. Skip-guards in Forge's deterministic gates so the framework's own
   `verify.sh` / `constitution-linter.sh` do not recurse into the
   example tree (the example owns its own gates).
4. A new `example` job in Forge's own `forge-ci.yml` that, on PRs
   touching `examples/**`, exercises the example tree.
5. Real measured baselines for the NFRs declared by `b1-delivery` тАФ
   `NFR-013`, `NFR-014`, `NFR-015`, `NFR-017` lose their "aspirational"
   status by gaining a measured "Baseline at archive time" line.

The FRs of this change are namespaced :

- `FR-GL-026..028` тАФ Forge-meta cross-cutting (skip-guards, gitignore).
- `FR-EX-001..010` тАФ example tree contract (new namespace dedicated to
  `examples/`).
- `FR-CI-012..013` тАФ Forge's own CI extensions for the example tree.

### FR-GL-026: Skip-guard for `examples/` in `verify.sh`

<!-- New in c1-reference-project (2026-04-29) -->

- **MUST** тАФ `.forge/scripts/verify.sh` detects when an inspected change
  directory or layer subtree falls under `<FORGE_ROOT>/examples/` and
  SKIPS it with an explicit `(skipped тАФ examples/ is self-contained)`
  message. The skip is gated on `$FORGE_ROOT` matching the Forge repo
  root (i.e. when the script runs **inside** the Forge framework
  repository, not when it runs inside a scaffolded example project).
- **MUST** тАФ the skip applies to every section that walks the file
  tree : `Change Artifact Completeness`, the multi-root layer scoping
  for `backend/` / `frontend/` / `infra/`, and the integration of
  `validate-foundations.sh`. The skip MUST NOT silently bypass other
  Forge-managed paths.
- **MUST** тАФ when invoked from inside `examples/forge-fsm-example/`
  (i.e. `FORGE_ROOT` resolves to the example tree itself), the skip
  guard does NOT activate ; the example owns its own `verify.sh` and
  exercises it standalone.
- **SHALL** тАФ emitted lines are prefixed `[skipped: examples] ...` for
  unambiguous aggregated output (consistent with the `b1-workflow`
  layer-scoping convention from FR-BE-002 / FR-FE-002).

**Constitution reference:** Article V (gates), Article X
(reproducibility). **Testable:** yes тАФ `test_verify_skips_examples_tree`
(positive : Forge repo with examples/ тЖТ emits skip lines) and
`test_verify_runs_inside_example_tree` (negative : invoked from inside
the example tree тЖТ no skip).

<!-- MODIFIED by c1-reference-project (2026-04-29) тАФ workflow now declares 6 top-level jobs (added `example`); summary's `needs:` list extended accordingly. -->

> The workflow declares **exactly five top-level jobs** : `harness`,

### FR-CI-006: Summary job aggregates the five into one required status

<!-- MODIFIED by c1-reference-project (2026-04-29) тАФ summary's `needs:` list extended to include the new `example` job. -->

**Previously:**
> `jobs.summary` declares `needs: [harness, gates, cli, lint]`.
> [...] Its bash step inspects each `needs.<job>.result` (read
> via `env:` indirection from `${{ needs.<job>.result }}`) and
> exits 1 if any is not exactly `'success'`. Skipped or
> cancelled upstreams MUST cause the summary to fail.

**Now:**
> `jobs.summary` declares `needs: [harness, gates, cli, lint,
> example]`. The bash step inspects each `needs.<job>.result`
> via `env:` indirection. Exit semantics : the four core jobs
> (`harness`, `gates`, `cli`, `lint`) MUST exit `'success'` ;
> the `example` job MUST exit `'success'` OR `'skipped'` (a
> paths-filter miss is not a failure). Any other result on any
> job тАФ `failure`, `cancelled`, or `skipped` on the four core
> jobs тАФ causes the summary to exit 1. The notice / error
> messages emitted (`forge-ci: 5/5 jobs PASS` or `forge-ci:

## Open Questions resolved at spec time

The proposal listed three open questions deferred to the design
phase. Two are resolved at spec time ; one remains for design.

### Q1 тАФ Demo-004 status (RESOLVED at spec time)

**Resolution:** demo-004 ships at status `specified` only. The
design phase is illustrated by the 3 archived demos
(`demo-001..003`) which already ship complete `design.md` files.

showing what `[NEEDS CLARIFICATION]` looks like in practice".
Codified in **FR-EX-005**.

### Q2 тАФ Example workflow execution method (RESOLVED at spec time)

**Resolution:** the new `example` job in `forge-ci.yml` does
**not** invoke `act` or otherwise execute the archetype reference
workflows. It performs a **structural YAML parse + integrity
check** of the `.tmpl` files (FR-CI-012 step 4) and exercises the
example's own `verify.sh` + `constitution-linter.sh` (FR-CI-012
steps 2 + 3). Real workflow execution is the job of the
example's own GitHub Actions, not Forge's CI. Codified in
**FR-CI-012**.

### Q3 тАФ Mono-change vs split (DECIDED by user тАФ kept as Q for design)

The user's pre-spec instruction "Garde le plan et ce qu'il faut
pour lancer l'option B + mono-change" closes this : single change,
not split. The change is shipped in three commit clusters
(scaffold / demos / docs+CI) for reviewer ergonomics, but stays
under the single `c1-reference-project` umbrella.
