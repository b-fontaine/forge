# Proposal: t5-connect-codegen
<!-- Created: 2026-05-05 -->
<!-- Schema: default -->

## Problem

`docs/ARCHITECTURE-TARGET.md` ADR-003 (REST/JSON Kong-bridge → Connect-RPC) and
ADR-009 (KEEP buf+proto + ADD Connect codegen + AsyncAPI 3.1 dérivé) ratify
**Connect-RPC as the target transport** for the flagship archetype. The plan
in `docs/new-archetypes-plan.md` §11 Phase 1 (T5) prescribes that Connect
codegen ship **additively, in parallel** to the existing tonic-build pipeline,
with **no removal** of the Kong-bridge REST API at this stage. This is the
designated first concrete action for T5 (§15 item #1, effort `L`).

The current `full-stack-monorepo / 1.0.0` template (shipped v0.3.0) has:

- A `buf.gen.yaml` configured for `tonic-build` (Rust gRPC server) and the
  REST/JSON bridge handled at the Kong layer.
- No Connect-RPC stubs generated for any layer (Rust server, TypeScript /
  Dart clients).
- No Connect-RPC demo in `examples/forge-fsm-example/` to validate the
  end-to-end traceparent W3C propagation through the new transport.
- No standard pinning the Connect codegen versions (the standard
  `transport.yaml` shipped in T4 names Connect-RPC as the default but does
  not yet specify `protoc-gen-connect-*` plugin versions or codegen layout).

Without this change, T6 (B.8 flagship 1.0.0 → 2.0.0 migration) cannot start:
the Envoy/DBOS/Connect bascule is the **point of no return**, and it
presupposes that Connect stubs already exist alongside the legacy
REST-bridge. T7 (B.6 `event-driven-eu` and B.7 `ai-native-rag`) also
presuppose buf + Connect as the contract foundation.

This is a **template/framework change** (zero adopter runtime code touched ;
adopters will pick up the new Connect stubs at next `forge upgrade` 3-way
merge, which leaves their Kong-bridge REST API intact).

## Solution

Single archived Forge change `t5-connect-codegen` that ships **four atomic
deliverables** :

1. **Connect codegen on the flagship template** — extend
   `templates/full-stack-monorepo/1.0.0/proto/buf.gen.yaml` with
   `protoc-gen-connect-go`, `protoc-gen-connect-es`, and
   `protoc-gen-connect-dart-community` plugin entries (versions pinned).
   Existing `tonic-build` invocation in the Rust workspace is **kept
   unchanged** (ADR-004). Output directories follow the layout prescribed
   by the `transport.yaml` standard (frontmatter pinning :
   `connect_layout_version: 1`).
2. **Connect-RPC server middleware on the Rust side** — add a parallel
   axum route that mounts the Connect handler generated by tonic +
   `protoc-gen-connect-go`-equivalent (Rust ecosystem : `tonic-web` +
   community Connect layer, OR `connect-rs` if mature enough). The
   existing tonic gRPC service stays mounted on its own port. Both are
   served behind Kong (which already proxies arbitrary HTTP ; no Kong
   plugin change required at this stage).
3. **Reference project demo `demo-005-connect-greeting`** added to
   `examples/forge-fsm-example/` : a single-layer backend change
   illustrating Connect-RPC end-to-end. The Greeter service from
   `demo-001-greeting-service` is exposed in parallel via Connect-RPC ;
   a TypeScript client (`web-public/`-shaped, but minimal — no Qwik
   dependency yet, that's T6) calls it ; traceparent W3C propagation is
   validated through the new path. Status : `archived` at change-archive
   time (full pipeline). NFR : the demo overlay must stay under 100 KB
   added to the example tree.
4. **`transport.yaml` extended** — bump `version: 1.0.0 → 1.1.0` (additive,
   no `breaking_change: true`), add a `codegen:` section pinning the
   three `protoc-gen-connect-*` plugin versions and the canonical output
   layout. `last_reviewed` stays at 2026-05-04 (still inside 12-month
   window, but `expires_at: never` per ADR-009 so this is informational).
   `REVIEW.md` ledger gets an `Updated` entry (not a full re-review).

The change is then archived and becomes the **transport foundation** for
T6 (B.8 flagship migration), T7 (B.6 + B.7), and T8 (B.9 mobile-pwa-first :
the Qwik client uses Connect-ES from this change).

## Scope In

- Extend `templates/full-stack-monorepo/1.0.0/proto/buf.gen.yaml` with
  three Connect codegen plugins (Go-equivalent for Rust workspace
  consumption, ES for TypeScript, Dart community for Flutter clients
  who later opt-in).
- Add a Connect-RPC server route on the Rust template
  (`templates/full-stack-monorepo/1.0.0/backend/src/`), mounted in
  parallel to the existing tonic gRPC service. Concrete crate decision
  documented in `design.md` (candidates : community `connect-rs`, or
  tonic + tower hand-rolled Connect codec).
- Generate stubs into a stable layout under `gen/connect/{rust,ts,dart}/`
  with `transport.yaml` `codegen.connect_layout_version: 1`.
- New reference demo `demo-005-connect-greeting` archived in
  `examples/forge-fsm-example/.forge/changes/`, with `proposal.md`,
  `specs.md`, `design.md`, `tasks.md`, BDD scenarios, and a working
  Connect-RPC handler + TypeScript caller.
- Validate traceparent W3C propagation : Flutter caller (existing
  demo-002 via tonic) → Kong → Rust ; new path : TypeScript caller →
  Kong → Rust Connect-RPC handler. Both must produce a single trace.
- Extend `transport.yaml` to `version: 1.1.0` with the new `codegen:`
  section (plugin versions pinned, layout version pinned).
- Append `Updated` entry to `.forge/standards/REVIEW.md` for transport.yaml.
- New test harness `t5.test.sh` (≥ 25 tests L1 hermetic + L2 fixture-based) :
  validates buf.gen.yaml parses, all three plugin entries present, layout
  contract holds, demo-005 archives cleanly, transport.yaml v1.1.0
  validates against its schema, end-to-end smoke test against the example.
- Update `docs/MIGRATION-PATHS.md` with a "T5 Connect codegen additive"
  section explaining what adopters get at next `forge upgrade` (and
  what stays untouched — the Kong-bridge REST API).
- Update snapshot tarball `full-stack-monorepo/1.0.0.tar.gz` (additive
  bump, no schema version change ; the `1.0.0` snapshot is updated in
  place, increasing snapshot size budget by ≤ 50 KB).
- Constitution-linter section `transport-codegen-coverage` (WARN-only at
  this stage) : warns if a flagship-derived project has `proto/` but no
  `gen/connect/` directories.

## Scope Out (Explicit Exclusions)

- **No Kong removal, no Envoy introduction** — Envoy Gateway is B.8 (T6).
  Kong stays as the only gateway in the flagship template. Connect-RPC
  traffic passes through Kong as plain HTTP.
- **No DBOS, no Zitadel, no Postgres+pgvector default** — those are B.8
  (T6) territory.
- **No Qwik public web** — Qwik client (B.8 / T6) and Qwik streaming
  UI (B.7 / T7) are out of scope. The TypeScript client in demo-005 is
  intentionally minimal (vanilla TS + Connect-ES, no framework).
- **No flagship schema bump 1.0.0 → 2.0.0** — this change is additive
  inside the 1.0.0 schema. The schema bump happens in B.8 (T6).
- **No new archetype** — B.6, B.7, B.9 are out of scope.
- **No agent refactor** — P-5 (Hera 9 → 5) stays deferred. No
  `.claude/agents/` modification.
- **No new agent** — Demeter (K.3), Hermes-Async (K.1), Pythia (K.2),
  Iris-Web (K.4), Themis (K.5) are out of scope. They land in T5 later
  items (K.3) or T7.
- **No flutter_bloc linter flip from WARN to ERROR** — that flip happens
  with B.8 (T6). The `no-state-management-alternatives` linter rule
  shipped in T4 stays WARN-only here.
- **No AsyncAPI 3.1 codegen** — ADR-009 mentions AsyncAPI dérivé from
  proto ; it ships with B.6 (`event-driven-eu`) in T7. Connect-only here.
- **No removal of REST/JSON Kong-bridge** — explicit ADR-003 phasing :
  Connect ships additively in T5, REST-bridge retired in B.8 (T6).
- **No Connect-Dart integration in `mobile-only` template** — B.9 (T8)
  brings Connect-ES to `mobile-pwa-first` ; the Dart Connect plugin is
  generated by `buf.gen.yaml` here for forward-compat but not consumed
  by any template in this change.

## Impact

- **Users affected** : all current adopters of `@sdd-forge/cli@0.3.0`
  who scaffolded a `full-stack-monorepo / 1.0.0` project. At next
  `forge upgrade`, they receive : the extended `buf.gen.yaml`, the new
  `gen/connect/` directories (empty until they re-run buf generate),
  and the optional Connect-RPC server route module (3-way merged ; if
  they have a hand-customised axum router, conflict markers expected
  but trivially resolvable). Their existing tonic gRPC + Kong-bridge
  REST stack is **untouched**.
- **Technical impact** : flagship template (`templates/full-stack-monorepo/1.0.0/`)
  + reference example (`examples/forge-fsm-example/`) + 1 standard
  (`transport.yaml` v1.1.0) + 1 review ledger entry + 1 test harness
  + 1 snapshot tarball update + linter section. **Complexity : L.**
- **Dependencies** :
  - T4 deliverables (transport.yaml v1.0.0 baseline, REVIEW.md, archetype
    schema v2). All shipped 2026-05-05.
  - External : `buf` CLI ≥ 1.28 (already pinned by F.2 / B.1), Rust
    toolchain stable (already pinned), Node ≥ 20 (already pinned by
    `cli/.nvmrc`).
  - Connect-Rust crate maturity : **risk to be assessed in design.md**.
    If `connect-rs` is not production-ready, fallback to a hand-rolled
    Connect codec on top of tonic+tower (acceptable per ARCH §14
    caveat 2). Documented as `[NEEDS DECISION:]` in proposal stage,
    resolved during `/forge:specify` or `/forge:design`.
- **Risk level** : **Low to Medium**.
  - Low : the change is additive ; existing tonic + Kong-bridge stays
    untouched ; rollback is `git revert` of the change ;
    `forge upgrade` 3-way merge handles adopter conflicts.
  - Medium : Connect-Dart plugin is community-only (ARCH §14 caveat 2) ;
    if the plugin breaks at codegen time, demo-005 still works (it uses
    the TS plugin), but adopters who want Dart Connect later are
    blocked until upstream fixes. Mitigation : `transport.yaml` pins
    a known-good version + `t5.test.sh` includes a Dart codegen smoke
    test ; if the smoke test fails, the dart plugin entry is gated
    behind a feature flag in `buf.gen.yaml`.

## Constitution Compliance

### Article I — TDD

Every shipped artefact has a corresponding test in `t5.test.sh`. RED-GREEN-REFACTOR :

- L1 hermetic tests written first (validate `buf.gen.yaml` shape, plugin
  entries, layout contract) — RED.
- L2 fixture-based tests (run buf generate against a frozen `.proto`
  fixture, assert generated files exist + contain expected symbols) —
  RED.
- Implementation : extend buf.gen.yaml, add Rust route, demo-005 — GREEN.
- Refactor : consolidate into `transport.yaml` codegen section — REFACTOR.

The reference demo `demo-005-connect-greeting` itself ships with its
own TDD compliance (Greeter service tests, Connect-RPC handler tests,
TypeScript client integration test).

### Article II — BDD

User-facing feature : a developer running `forge init --archetype
full-stack-monorepo` then `buf generate` gets Connect stubs in
`gen/connect/`. BDD scenarios shipped in `demo-005-connect-greeting/specs.md` :

```
Scenario: Connect-RPC client successfully calls Greeter
  Given the Greeter service is running on localhost:50051 (gRPC)
    And the Connect-RPC route is mounted at /connect on localhost:8080
  When a TypeScript client sends a Connect Greet request
  Then the response contains the expected greeting
    And the trace propagation header (traceparent) is preserved end-to-end
```

### Article III — Specs Before Code

Confirmed : `specs.md` will ship with explicit `FR-T5-CC-NNN` IDs (Connect
Codegen functional requirements) plus `NFR-T5-*` for budgets (snapshot
tarball delta ≤ 50 KB, harness runtime ≤ 5 s, generated stub size ≤
500 KB for the demo proto). Implementation starts only after `specs.md`
and `design.md` are gate-cleared.

### Article IV — Delta Specifications

`specs.md` opens with `## ADDED` (Connect codegen plugins, Rust route,
demo-005, transport.yaml codegen section, t5.test.sh, linter rule),
`## MODIFIED` (transport.yaml v1.0.0 → v1.1.0, snapshot tarball
full-stack-monorepo/1.0.0), and `## REMOVED` (none — additive change).
Linter validates via F.4's existing rules.

### Article V — Constitution Gate

`/forge:design` and `/forge:review` re-run the gate. The change does
not introduce any Article violation by construction (additive,
spec-first, TDD, BDD covered).

### Article VI — Flutter Architecture

The Dart Connect plugin is generated for forward-compat but no Flutter
template (in this change) consumes it. `mobile-only / 1.0.0` template
is not touched. Article VI applies trivially.

### Article VII — Rust Architecture

Rust template gains a parallel Connect-RPC route. Hexagonal architecture
preserved : Connect handler is an adapter, the Greeter use case stays
in the domain layer. tonic gRPC adapter coexists with the new Connect
adapter. ADR-004 (KEEP Rust + tonic 0.14 + axum 0.8) respected.

### Article VIII — Infrastructure

No K8s, Helm, Docker change in the template's infra layer. Kong config
template untouched (Connect traffic is plain HTTP through Kong). Article
VIII inapplicable beyond this verification.

### Article IX — Security

No new secret material. No new external service. Connect-RPC inherits
the same Kong rate-limit and JWT validation plugins as the existing
REST-bridge. Aegis review optional (recommended, not blocking).

### Article X — Quality

`tasks.md` ships with explicit task ↔ FR-T5-CC-* linkage. Public API
doc ratio (Article X.3) : the new Rust Connect route module documents
all public functions (≥ 80 %).

### Article XI — AI-First

Not an AI feature. Article XI inapplicable.

### Article XII — Governance

Ratifies ADR-003 + ADR-009 (already ratified in t4-adr-ratification under
Constitution v1.1.0). No Constitution amendment. `constitution_version:
"1.1.0"` stays. Public discussion already covered by `docs/ARCHITECTURE-TARGET.md`
review and t4-adr-ratification archive.

## Open Questions

See `open-questions.md` (to be created during `/forge:specify`).

Pre-flagged decisions to resolve in specs / design phase :

- **Q-001** (design) : Which Connect-Rust crate? `connect-rs` if mature,
  otherwise hand-rolled tonic+tower. Acceptance criteria : production-
  ready, OTel middleware compatible, MIT/Apache-licensed.
- **Q-002** (specs) : Plugin version pins for `protoc-gen-connect-{go,es,dart-community}`.
  Pick the latest stable as of 2026-05-05 ; pin in `transport.yaml`.
- **Q-003** (design) : Should demo-005 ship a TypeScript client only, or
  also a minimal Rust Connect client (proving server-to-server)? Default
  proposal : TS only, keep scope tight.
- **Q-004** (specs) : Layout convention for `gen/connect/{rust,ts,dart}/` —
  flat or nested per-service. Pin in `transport.yaml` `codegen.connect_layout_version: 1`.
