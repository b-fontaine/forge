## Problem

`b1-foundations` landed the **contract** for `full-stack-monorepo`: a
schema, three standards, scoped commits, monorepo versioning, and a
deterministic validator. What's missing is the **generator** that
produces a new project that conforms to that contract.

Without this change :

- No user can run `/forge:init --archetype full-stack-monorepo` — the
  slash command has no branch for the archetype flag (it currently only
  handles the copy-from-assets path).
- `b1-workflow` (agent Janus + multi-root linters + multi-layer change
  workflow) cannot be built on anything — it needs a scaffolded sample
  project to orchestrate against.
- `b1-delivery` (CI workflows + `forge upgrade` + env matrix) has no
  reference project structure to target.
- Audit module **C.1** (first public reference project scaffolded via the
  archetype) is blocked — it depends on this change by definition.

The contract from `b1-foundations` prescribes exact shapes
(`layers: [backend, frontend, infra]`, forbidden cross-imports, nested
`CLAUDE.md`, scoped commits, `fsm-*` service naming, `shared/protos/`
as single source of truth). `b1-scaffolder` materializes every one of
these shapes into generated starter files, so the scaffolded project
passes `validate-foundations.sh` out of the box.

## Solution

Ship four tightly-related deliverables, collectively sufficient for
`/forge:init --archetype full-stack-monorepo` to produce a project that
satisfies the `full-stack-monorepo` schema and every standard it cites:

1. **Archetype template tree** (`B.1.2`, `B.1.3`, `B.1.4`, `B.1.13`) —
   under `.forge/templates/archetypes/full-stack-monorepo/`, a static
   file tree that the scaffolder copies verbatim into the target project
   (with trivial substitutions for `<project-name>`, `<reverse-domain>`,
   `<root-module>`). Contents :

    - Root : `CLAUDE.md` (routing-only, no stack-specific standards),
      `Taskfile.yml` (canonical targets: `dev`, `test`, `lint`, `proto`,
      `release`, plus per-layer aliases `test:backend`, `test:frontend`,
      `test:infra`), `docker-compose.dev.yml` (the three canonical
      services `fsm-backend`, `fsm-kong`, `fsm-db` with healthchecks +
      the `fsm-dev` network + `env_file: .env`), `.env.example`,
      `.gitignore`, `.forge.yaml` with `schema: full-stack-monorepo`,
      `README.md` seeded from an archetype-specific template.
    - `frontend/CLAUDE.md`, `backend/CLAUDE.md`, `infra/CLAUDE.md` —
      nested CLAUDE.md that scope the standards loaded when Claude Code
      navigates the subtree (FR-GL-002 in the contract).
    - `shared/protos/buf.yaml`, `shared/protos/buf.gen.yaml`,
      `shared/protos/v1/example/example.proto` — a minimal starter that
      `buf lint` + `buf breaking` accept.
    - `backend/Cargo.toml` (workspace manifest), `backend/rust-toolchain.toml`,
      `backend/.cargo/config.toml` (optional) — the workspace root.
      Actual crate content is created by the official scaffolder
      (`cargo new`, see step 3).
    - `infra/kong/kong.yml.example`, `infra/k8s/base/.gitkeep`,
      `infra/docker/Dockerfile.backend.example`, minimal stubs for each
      subtree.

2. **Scaffolder logic in `/forge:init` slash command** (`B.1.2`) — the
   file `.claude/commands/forge/init.md` gains a dedicated branch for
   `--archetype full-stack-monorepo`. Sequence (non-negotiable order, per
   `monorepo-layout.md` rule "official scaffolder first, Forge overlay
   second") :

    1. Create the target directory, `cd` into it.
    2. `flutter create frontend --org <reverse-domain> --platforms
       android,ios,web --project-name <project_name>_frontend` — the
       official Flutter scaffolder. Fail the command if `flutter` is not
       on PATH ; do NOT fall back to a hand-rolled skeleton.
    3. `cargo new --vcs none backend` + `cd backend && cargo new --lib
       crates/domain --vcs none && cargo new --lib crates/application
       --vcs none && cargo new --lib crates/grpc-api --vcs none &&
       cargo new --lib crates/infrastructure --vcs none && cargo new
       bin-server --vcs none` — the official Cargo workspace scaffolder.
    4. Overlay the archetype templates on top of the scaffolded tree
       (NEVER touches files generated by `flutter create` / `cargo new`
       except when an explicit override is documented in the template
       with a header comment).
    5. `cd shared/protos && buf lint && buf breaking --against
       '.git#branch=main' || true` — first invocation of the proto
       gates ; the `|| true` on `buf breaking` is acceptable on the
       seed commit because `main` does not yet exist.
    6. Run `validate-foundations.sh` (from `b1-foundations`) against the
       fresh target to certify the scaffold. Fail if any check fails.

3. **Scaffold plan descriptor** (`B.1.2`) — a machine-readable YAML
   file `.forge/templates/archetypes/full-stack-monorepo/scaffold-plan.yaml`
   that lists each template file, its substitution keys, and its target
   relative path. The slash command reads this plan instead of hard-coding
   file names, and a future contributor can add a file to the archetype
   by editing the plan alone. This also unblocks a future CLI extension
   (not in scope here) that could consume the same plan.

4. **Integration test** (`B.1.2`, FR-GL-008 continuation) — a new test
   harness `.forge/scripts/tests/scaffolder.test.sh` that:
    - Runs the scaffolder sequence in a `mktemp -d` target.
    - Asserts every mandatory file exists at the expected relative path.
    - Runs `validate-foundations.sh` against the scaffolded target and
      asserts `exit 0` with 7 PASS lines.
    - Asserts that `flutter create` and `cargo new` markers (files
      generated by the official scaffolders) are present and unmodified.
    - Is invoked by `foundations.test.sh` as a new `run_test` entry OR
      lives as an independent top-level harness (design decision —
      ADR in `/forge:design`).

## Scope In

- Archetype template tree (item 1 above), complete for a minimally
  buildable project.
- `/forge:init` slash command enrichment with the `--archetype
  full-stack-monorepo` branch (item 2).
- Machine-readable scaffold plan descriptor (item 3).
- Integration test harness for the scaffolder (item 4), covered by the
  existing `foundations.test.sh` runner.
- Documentation: `README.md` seed for the archetype explaining the
  generated layout ; cross-links from `global/monorepo-layout.md`
  pointing at the scaffolder command.

## Scope Out (Explicit Exclusions)

- **Agent Janus** (`B.1.7`) — reported in `b1-workflow`. The scaffolded
  CLAUDE.md files reference Janus by name but never invoke him.
- **Multi-root `verify.sh` / `constitution-linter.sh`** (`B.1.8`) —
  reported in `b1-workflow`. The scaffolded project uses the existing
  single-root scripts ; they return useful results thanks to the
  canonical layout.
- **CI workflows** (`B.1.9`) — reported in `b1-delivery`. The scaffolded
  `.github/workflows/` directory is intentionally empty with a
  `.gitkeep`.
- **`forge upgrade` command** (`A.7`) — separate change, not tied to
  Module B.
- **Env overlays** (`B.1.12`) and **observability integration**
  (`B.1.14`) — reported in `b1-delivery`. Scaffolded `infra/k8s/overlays/`
  is empty with a `.gitkeep`. The dev compose stack does NOT include the
  OTel collector in this change.
- **Taskfile targets beyond the canonical five** — e.g. `task bench`,
  `task release:candidate`, `task docs`. Only `dev`, `test`, `lint`,
  `proto`, `release` and their per-layer aliases ship here.
- **Extending `@sdd-forge/cli`** (the TypeScript CLI) with an
  `--archetype` flag — out of scope. The slash command orchestrates
  the shell invocations ; the CLI stays a pure asset-copier. A future
  change may port the logic into the CLI for non-Claude-Code harnesses.
- **No modification of the Constitution** — the archetype is an
  additive extension that lives within the existing articles.

## Impact

- **Users affected** :
    - Forge adopters who want to start a new `full-stack-monorepo`
      project — they get a one-command bootstrap.
    - Contributors writing `b1-workflow` and `b1-delivery` — they have
      a reference project structure to target.
    - The planned reference project (`C.1`) — directly unblocked.
- **Technical impact** :
    - Files created : ~25 template files under
      `.forge/templates/archetypes/full-stack-monorepo/` + 1 scaffold
      plan YAML + 1 integration test script.
    - Files modified : 1 (`.claude/commands/forge/init.md` — enriched
      with archetype branch) ; 1 (`.forge/scripts/tests/foundations.test.sh`
      — adds scaffolder integration test invocation).
    - Runtime surface changed : a scaffolded project acquires its own
      concrete layout ; nothing in an existing Forge project is
      affected.
    - Complexity : **L** (Large). Orchestrating official scaffolders +
      overlay logic + proto toolchain is non-trivial.
- **Dependencies** :
    - Upstream : `b1-foundations` archived. The schema and standards
      are consumed directly by the templates and the scaffold plan.
    - External : `flutter` ≥ 3.24 and `cargo` ≥ 1.80 on the user's
      PATH. `buf` CLI is required at run time but installed on demand
      by the scaffolded `Taskfile.yml`.
    - Downstream : `b1-workflow`, `b1-delivery`, Module C.1.
- **Risk level** : **Medium-High**. Primary risks :
    - **Version drift of external scaffolders** (Flutter SDK, Cargo) —
      mitigated by NOT pinning versions in the overlay ; we record the
      generation timestamp in the scaffolded project's README for
      auditability.
    - **Silent divergence** between scaffolded output and the contract —
      mitigated by running `validate-foundations.sh` as the last step
      of scaffolding and failing on any FAIL.
    - **External network dependency** (buf tool download) — mitigated
      by documenting installation in the README and making `buf lint`
      a WARN (not FAIL) on the first run if buf is absent, with a
      hint to install it.

## Constitution Compliance

### Article I — TDD

The deliverables are a mix of templates (pure content) and orchestration
logic (the slash command plus the integration test script). The TDD
cycle is applied to the test harness :

1. RED : write `scaffolder.test.sh` that asserts (a) every mandatory
   file exists at the expected relative path, (b)
   `validate-foundations.sh` returns PASS on the scaffolded target,
   (c) official-scaffolder markers are preserved. The test fails
   because no templates and no slash-command branch exist yet.
2. GREEN : write the minimum templates and the minimum slash-command
   branch that make each assertion pass, one at a time.
3. REFACTOR : extract any duplicated template content into shared
   partials ; ensure the scaffold plan is the single source of truth
   for "what gets generated".

### Article II — BDD

A user-facing feature (a developer running `/forge:init --archetype
full-stack-monorepo`) exists — so a `features/b1-scaffolder.feature`
file is REQUIRED. Scenarios :
- Happy path : developer with Flutter + Cargo on PATH scaffolds a new
  project ; the validator passes.
- Missing tool : developer without Flutter on PATH ; scaffolder fails
  early with a clear error.
- Existing directory : developer scaffolds into a non-empty directory ;
  scaffolder refuses unless `--force` is provided.
- Validator regression : after scaffold, modify one file so the
  validator fails ; re-running the scaffolder with `--force` fixes it.

### Article III — Specs Before Code

Confirmed. `/forge:specify b1-scaffolder` runs next.

### Article IV — Delta Specs

`specs.md` uses ADDED / MODIFIED / REMOVED. Additive only — no FR from
`b1-foundations` is modified or removed.

### Article VI / VII — Architecture (Flutter / Rust)

The templates MUST generate a Flutter `frontend/` that respects FSD
(feature-sliced) and Clean Architecture (Article VI.2), and a Rust
`backend/` workspace that respects Hexagonal Architecture
(Article VII.3). The crate layout `domain / application / grpc-api /
infrastructure / bin-server` directly encodes the hexagonal boundaries.

### Article VIII — Infra

`docker-compose.dev.yml` conforms to `infra/docker-compose.md`
(fsm-* naming, healthchecks, `.env.example`). `Dockerfile.backend.example`
is a multi-stage distroless template per Article VIII.3.

### Article IX — Observability / Security

The scaffolded project ships WITHOUT observability wiring in this change
(deferred to `b1-delivery` B.1.14). `.env.example` stubs make it clear
that no real secret lives in the repository.

### Article X — Quality

Every template file carries the `<!-- Audit: B.1.X -->` header
(or YAML / Shell equivalent) per NFR-004 from `b1-foundations`.
Markdownlint is applied to every Markdown template.

### Article XI — AI-First

N/A — no AI feature in this change.

## Open Questions

Decisions to be resolved during `/forge:specify` or `/forge:design` :

- `[NEEDS CLARIFICATION: should `backend/Cargo.toml` be a workspace
  manifest with the 5 crates enumerated from the start, or should the
  scaffolder generate them one-by-one via `cargo new` and then edit
  `Cargo.toml` to add the `members` list?]` — the two approaches differ
  in how "official-scaffolder-first" is interpreted at the workspace
  level. Resolution proposed : generate a minimal workspace
  `Cargo.toml` in advance (just `[workspace]` with an empty `members`
  list), then use `cargo new --lib crates/<name>` for each crate which
  will auto-append to `members` IF the `cargo new` heuristic detects
  the parent workspace (behaviour verified in Cargo ≥ 1.75). Document
  the assumption in the template and provide a fallback manual edit.

- `[NEEDS CLARIFICATION: should the scaffolder require `--target-dir`
  explicitly, or scaffold into the current working directory by
  default?]` — precedent from `flutter create` and `cargo new` is to
  take a positional directory argument. Proposed : require a positional
  `<project-name>` argument ; scaffold into `./<project-name>/`. No
  default of "cwd" to avoid accidental pollution of an existing repo.

- `[NEEDS CLARIFICATION: proto starter — should we scaffold a real
  example service (e.g. `UserService` with `SignIn` / `SignOut` RPCs)
  or a trivial `example.proto` with one message and no RPC?]` —
  proposed : trivial `example.proto` with a single `Ping` RPC under
  `examples.v1`. Just enough to make `buf lint` succeed and the
  codegen toolchain exercisable. Avoids prescribing domain concepts
  that would date the archetype.

These three questions do not block the specification phase —
`/forge:specify` will lock them via explicit FRs.
